Zoop receives inbound webhooks from three providers: Stripe (payments), Twilio (SMS), and Resend (email). These endpoints are public — no `Authorization` header is required — but every request is rejected unless the provider's signature is valid. You do not call these endpoints yourself; the providers do.

There are no outbound webhooks for third-party developers to subscribe to. Zoop does not deliver event notifications to URLs you register — there is no subscription mechanism, and every webhook route Zoop exposes is inbound only. If you need to react to events in Zoop, poll the API.

---

## How verification works

Each provider uses a different scheme.

| Provider | Method | Key header(s) |
|---|---|---|
| Stripe | HMAC-SHA256 over the raw body | `Stripe-Signature` |
| Twilio | HMAC-SHA1 over the canonical request URL + params | `X-Twilio-Signature` |
| Resend | Svix (standardwebhooks HMAC) over the raw body | `svix-id`, `svix-timestamp`, `svix-signature` |

A request that fails signature verification is rejected with a `400` or `401` response depending on the provider. The provider will retry — Zoop does not silently drop unverified bodies.

## Stripe

Zoop registers two separate Stripe webhook destinations. Each has its own signing secret and handles a different scope of events.

### Stripe platform

```http
POST /api/webhooks/stripe/platform
```

Receives events on the **Your account** scope — events that happen on the Zoop platform account itself (platform-level charges, indirect destination charges, and Stripe Connect v2 account state changes).

### Stripe connect

```http
POST /api/webhooks/stripe/connect
```

Receives events on the **Connected accounts** scope — events that happen on your Stripe connected account (charges, refunds, disputes, payouts, and related objects).

### Raw body requirement

<Callout type="warning">
Stripe computes its HMAC against the **exact raw bytes** it sent. If anything re-serializes the body before verification — for example, parsing JSON and then stringifying it — the signature will not match and the request will be rejected. Always read the body as a raw string; never call `request.json()` before verifying.
</Callout>

### Verification flow

Both Stripe routes follow the same pattern:

<Steps>
  <Step title="Read the raw body">
    The route reads the request body as a plain string using `request.text()` — not `request.json()`.
  </Step>
  <Step title="Verify the Stripe-Signature header">
    The signing secret for the endpoint is read from an environment variable. If the signature does not match, the route returns `400 { "error": "Invalid signature" }` and Stripe retries.
  </Step>
  <Step title="Check live/test mode">
    The event's `livemode` field is compared against the deployment's expected mode (derived from the `STRIPE_SECRET_KEY` prefix). A mismatch parks the event and returns `200` — it does not make Stripe retry, because the mismatch is a misconfiguration, not a transient error.
  </Step>
  <Step title="Ingest atomically">
    Verified events are written to the `domain_event_outbox` table in a single transaction. If a Stripe event ID was already ingested (re-delivery), the route responds `200 { "received": true, "deduped": true }` and skips the write.
  </Step>
</Steps>

### Stripe response shapes

```json
// Normal ingest
{ "received": true, "ingested": 1 }

// Re-delivered event (already ingested)
{ "received": true, "deduped": true }

// Event parked due to live/test mode mismatch
{ "received": true, "parked": true, "reason": "mode_mismatch" }

// No parseable event ID — acknowledged, not ingested
{ "received": true, "ingested": 0 }
```

### Stripe error responses

| Status | Body | Meaning |
|---|---|---|
| `400` | `{ "error": "Invalid signature" }` | Signature check failed. Stripe will retry. |
| `500` | `{ "error": "ingest_failed" }` | Database write failed. Stripe will retry. |
| `500` | `{ "error": "misconfigured" }` | Signing secret env var is missing. Stripe will retry while you fix config. |

## Twilio

Zoop registers two Twilio webhook callbacks: one for messages your customers send in (inbound SMS), and one for delivery status updates on messages Zoop sends out.

### Inbound SMS

```http
POST /api/webhooks/twilio/inbound-sms
```

Called by Twilio when a customer sends an SMS to a Zoop-managed phone number. The request body is `application/x-www-form-urlencoded`.

**Required fields Twilio sends:**

<ParamField body="MessageSid" type="string" required>
  Twilio's unique identifier for the message (e.g. `SMxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`).
</ParamField>

<ParamField body="From" type="string" required>
  The sender's phone number in E.164 format (e.g. `+15558675309`).
</ParamField>

<ParamField body="To" type="string" required>
  The Zoop-managed phone number that received the message. Used to look up which tenant owns this number.
</ParamField>

<ParamField body="Body" type="string">
  The SMS message text. Defaults to an empty string if absent.
</ParamField>

**What happens after verification:**

1. The `To` number is looked up against `tenant_inbound_addresses`. If no tenant owns it, the message is quarantined (stored for operator review) and Twilio receives a `200`.
2. If a tenant is found, Zoop tries to match the `From` number to an existing customer and contact.
3. The message is written to the `communications` table with `channel: 'sms'`, `direction: 'inbound'`, `status: 'received'`.
4. A duplicate `MessageSid` (Twilio re-delivery) is silently acknowledged with `200` — the unique index on `(tenant_id, provider, provider_message_id)` prevents double-writes.

**Response:** Always `text/xml` TwiML — `<Response></Response>` (empty, no reply). Zoop responds `200` on success and `401` on signature failure. A `5xx` response causes Twilio to retry.

### SMS status callback

```http
POST /api/webhooks/twilio/sms-status
```

Called by Twilio when the delivery status of an outbound SMS changes. The request body is `application/x-www-form-urlencoded`.

**Required fields Twilio sends:**

<ParamField body="MessageSid" type="string" required>
  The same SID Zoop received when the message was sent.
</ParamField>

<ParamField body="MessageStatus" type="string" required>
  The new status. Zoop maps the following Twilio values:

  | Twilio status | Zoop maps to |
  |---|---|
  | `accepted`, `scheduled`, `queued` | `queued` |
  | `sending`, `sent` | `sending` |
  | `delivered` | `delivered` |
  | `undelivered`, `failed` | `failed` |
  | `read`, `receiving`, `received` | no change |
</ParamField>

<ParamField body="ErrorMessage" type="string">
  Human-readable failure reason. Stored on the row when status becomes `failed`.
</ParamField>

<ParamField body="ErrorCode" type="string">
  Twilio error code. Stored on invoice delivery rows when status becomes `failed`.
</ParamField>

**What happens after verification:**

Zoop looks up the `MessageSid` in the `communications` table first, then falls back to `invoice_deliveries`. A state-machine guard prevents backwards transitions and replays (e.g., a `queued` callback arriving after `delivered` is ignored). The route always responds `200` so Twilio does not retry stale callbacks.

### Twilio signature verification detail

Twilio signs the **exact public URL** it posted to plus all form parameters. On Vercel, the internal hostname may differ from the public hostname, so Zoop reconstructs the canonical public URL from trusted proxy headers (`x-forwarded-*`) before computing the signature check.

## Resend

Resend uses [Svix](https://www.svix.com) (an implementation of the standardwebhooks spec) to deliver and sign webhook payloads. Zoop registers two Resend endpoints.

### Resend inbound email

```http
POST /api/webhooks/resend/inbound
```

Called by Resend when a customer sends an email to a Zoop-managed address. The request body is JSON, signed with the `RESEND_WEBHOOK_SECRET` env var.

**Svix headers required:**

<ParamField header="svix-id" type="string" required>
  Unique delivery ID. Used by Svix for replay protection.
</ParamField>

<ParamField header="svix-timestamp" type="string" required>
  Unix timestamp of the delivery. Used in the HMAC calculation.
</ParamField>

<ParamField header="svix-signature" type="string" required>
  The HMAC signature string. Format: `v1,<base64-encoded-hmac>`.
</ParamField>

**Payload fields:**

<ParamField body="from" type="string" required>
  Sender's email address.
</ParamField>

<ParamField body="to" type="string" required>
  The Zoop-managed email address that received the message. Used to look up the tenant.
</ParamField>

<ParamField body="messageId" type="string" required>
  Resend's unique message ID.
</ParamField>

<ParamField body="subject" type="string">
  Email subject line.
</ParamField>

<ParamField body="text" type="string">
  Plain-text body of the email.
</ParamField>

<ParamField body="html" type="string">
  HTML body of the email.
</ParamField>

<ParamField body="inReplyTo" type="string">
  `In-Reply-To` header value, used for threading the message into an existing conversation.
</ParamField>

<ParamField body="dmarc" type="string">
  DMARC verdict from Resend's gateway. One of `pass`, `fail`, `neutral`, or `none`. When `fail`, Zoop persists the message but does not correlate it to a customer (the `From` header is untrusted).
</ParamField>

<ParamField body="attachments" type="array">
  Array of attachment objects. Each has `filename` (string), `content` (base64 string), and `contentType` (string).

  Zoop enforces these limits before storing:
  - Per-attachment max: 10 MB
  - Total per-message max: 25 MB
  - Max attachment count: 20
  - Allowed MIME types: `image/jpeg`, `image/png`, `image/gif`, `image/webp`, `application/pdf`, `text/plain`, and common Microsoft Office and OpenDocument formats. HTML and other types are rejected.
</ParamField>

**What happens after verification:**

1. The `to` address is resolved to a tenant. Unmatched addresses are quarantined and Resend receives `200`.
2. The `communications` row is inserted **before** any attachment uploads. A duplicate `messageId` (Svix re-delivery) short-circuits at the unique index and returns `200` without re-uploading.
3. Attachments are uploaded to Supabase Storage (`communications-attachments` bucket) and their paths are written back to the row. If an upload fails, the text body is still preserved.
4. Svix retries on `5xx`. A `200` with `accepted: false` is returned when the signature is valid but the payload schema is unrecognized (provider drift) — retrying would not help.

### Resend email events

```http
POST /api/webhooks/resend/email-events
```

Called by Resend when the delivery status of an outbound email changes. Signed the same way as the inbound endpoint (Svix, same `RESEND_WEBHOOK_SECRET`).

**Payload fields:**

<ParamField body="type" type="string" required>
  The event type. One of:

  | Event | Zoop maps to |
  |---|---|
  | `email.sent` | `sending` |
  | `email.delivered` | `delivered` |
  | `email.opened` | `delivered` (also sets `read_at`) |
  | `email.clicked` | `delivered` (also sets `read_at`) |
  | `email.bounced` | `failed` |
  | `email.complained` | `failed` |
  | `email.failed` | `failed` |
  | `email.delivery_delayed` | no change (informational) |
</ParamField>

<ParamField body="data.email_id" type="string" required>
  Resend's message ID. Matched against `provider_message_id` in `communications` and `provider_id` in `invoice_deliveries`.
</ParamField>

**What happens after verification:**

Zoop looks up the `email_id` in `communications` first, then falls back to `invoice_deliveries`. The route always responds `200` — even on a database update failure — to prevent Svix from triggering a ~24-hour retry storm on a non-transient error.

## Testing locally

You can forward provider webhooks to your local dev server using each provider's official CLI.

<Tabs>
  <Tab title="Stripe">
    ```bash
    stripe listen --forward-to localhost:3000/api/webhooks/stripe/connect
    ```

    The Stripe CLI prints a signing secret (`whsec_...`) on startup. Set it as `STRIPE_WEBHOOK_SECRET_CONNECT_TEST` in your `.env.local` (the env var name carries both the destination — `CONNECT` or `PLATFORM` — and the mode — `TEST` or `LIVE`). To test the platform destination instead, forward to `/api/webhooks/stripe/platform` and set `STRIPE_WEBHOOK_SECRET_PLATFORM_TEST`.
  </Tab>
  <Tab title="Twilio">
    ```bash
    # Expose your local server with a tunnel first (e.g. ngrok)
    ngrok http 3000

    # Then point your Twilio phone number's webhook URL at:
    # https://<your-ngrok-subdomain>.ngrok-free.app/api/webhooks/twilio/inbound-sms
    ```

    Twilio signs requests using your `TWILIO_AUTH_TOKEN`. Make sure your local `.env.local` has the same auth token as the Twilio account sending the test request.
  </Tab>
  <Tab title="Resend">
    ```bash
    # Expose your local server with a tunnel first
    ngrok http 3000

    # Then register the tunnel URL as a Resend webhook endpoint in the
    # Resend dashboard. Resend provides a signing secret (whsec_...) —
    # set it as RESEND_WEBHOOK_SECRET in your .env.local.
    ```
  </Tab>
</Tabs>

### Stripe webhook signing secret env vars

Stripe issues a distinct signing secret per endpoint per mode, so Zoop reads a different env var for each of the four combinations. Zoop derives the mode (test or live) from the prefix of `STRIPE_SECRET_KEY` (`sk_test_`/`rk_test_` is test, `sk_live_`/`rk_live_` is live) and then reads the matching secret:

| Destination | Mode | Env var |
|---|---|---|
| Connect | test | `STRIPE_WEBHOOK_SECRET_CONNECT_TEST` |
| Connect | live | `STRIPE_WEBHOOK_SECRET_CONNECT_LIVE` |
| Platform | test | `STRIPE_WEBHOOK_SECRET_PLATFORM_TEST` |
| Platform | live | `STRIPE_WEBHOOK_SECRET_PLATFORM_LIVE` |

For the Connect destination only, Zoop falls back to the pre-rename names `STRIPE_WEBHOOK_SECRET_TEST` / `STRIPE_WEBHOOK_SECRET_LIVE`, and then to the legacy single-secret `STRIPE_WEBHOOK_SECRET`, if the new per-mode var is unset. The Platform destination has no fallback — you must set its per-mode var explicitly. If the resolved secret is missing, the route returns `500 { "error": "misconfigured" }` and Stripe retries while you fix config.

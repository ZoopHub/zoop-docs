Zoop's developer surface is an [MCP](https://modelcontextprotocol.io) (Model Context Protocol) server. You send JSON-RPC 2.0 requests to a single HTTP endpoint and get back structured data about a Zoop tenant — customers, jobs, invoices, quotes, and more. This is the same tool layer the in-product Zoop AI agent uses, exposed as a stable API for your own integrations.

<Callout type="info">
There is no OpenAPI/REST spec yet. All programmatic access goes through the MCP endpoint described here.
</Callout>

## What you can build

A few examples of what the API makes possible:

- **AI agents and chatbots** — connect Claude Desktop, LangGraph, n8n, or any MCP-capable agent framework to a contractor's Zoop account so it can look up customers, create jobs, draft invoices, and more.
- **Reporting dashboards** — pull jobs, invoices, and quotes into your own data warehouse or BI tool.
- **Automation scripts** — create jobs or notes in bulk, sync customers from an external CRM, or trigger follow-up tasks when a job is marked done.
- **Internal tools** — build a custom dispatch interface or field-tech mobile view on top of the same data Zoop uses.

## The API surface

**Endpoint:** `POST /api/mcp`

**Protocol:** MCP over streamable HTTP transport, JSON-RPC 2.0.

There is one endpoint. You do not need a tenant ID in the URL — the tenant a request operates on is bound to your access token, not the path.

The server advertises **65 tools** across **13 entities**:

| Entity | Tools include |
|---|---|
| Customers | search, get, create, update, archive |
| Contacts | list, get, create, update, archive |
| Locations | list, get, create, update, archive |
| Tags | list, get, create, update, delete |
| Notes | list, get, create, update, archive |
| Jobs | list, get, create, update, cancel |
| Job series | list, get, create, update, end |
| Invoices | list, get, create, update, void |
| Quotes | list, get, create, update, archive |
| Catalog items | list, get, create, update, archive |
| Catalog categories | list, get, create, update, delete |
| Plans | list, get, create, update, cancel |
| Tax rates | list, get, create, update, archive |

Every entity follows the same five-verb shape: list or search, get one, create, update (partial patch), and a destructive verb. The full catalog with scope requirements is in [MCP tools](/developers/mcp).

## Authentication

Every request needs an `Authorization: Bearer <token>` header. Two token types are accepted:

<Columns>
<Card title="API keys" href="/developers/authentication/api-keys">
The simplest path. Issue a key from **Settings → API & MCP** in Zoop (owner role required). Pass it directly as the bearer token — no token exchange needed. Use this for server-side scripts, background jobs, and agent pipelines.
</Card>
<Card title="OAuth 2.1" href="/developers/authentication/oauth">
For integrations where your users connect their own Zoop accounts. Implements OAuth 2.1 with PKCE. The issued JWT carries the tenant ID and scopes in its claims.
</Card>
</Columns>

### Key prefixes

When you issue an API key you choose the actor type:

| Prefix | Actor type | Has a user identity |
|---|---|---|
| `zoop_uk_` | User key | Yes — writes are attributed to the issuing user |
| `zoop_tk_` | Tenant key | No — reads and un-attributed writes only |

Tools that stamp `created_by` or require a real user identity are marked **U** in the tool catalog and can only be called with a user key or an OAuth token.

### Scopes

Both API keys and OAuth tokens carry **scopes** that restrict which tools they can call. Scopes follow a `read:<entity>` / `write:<entity>` shape — for example, `read:customers` or `write:jobs`. A token without the right scope gets a `403 insufficient_scope` response naming the missing scope.

The full scope list is in [Authentication — scopes](/developers/authentication/scopes).

## Making your first request

The example below calls `tools/list` to confirm your token works and see the available tools. Replace `zoop_uk_<secret>` with your actual API key.

```bash
curl -X POST https://app.zoop.example/api/mcp \
  -H "Authorization: Bearer zoop_uk_<secret>" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
```

A successful response returns a JSON-RPC envelope containing an array of tool definitions. If you get a `401`, your token is invalid or expired. If you get a `403`, check the scope.

To call a specific tool, change `method` to `tools/call` and add a `params` body:

```bash
curl -X POST https://app.zoop.example/api/mcp \
  -H "Authorization: Bearer zoop_uk_<secret>" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{
    "jsonrpc": "2.0",
    "id": 2,
    "method": "tools/call",
    "params": {
      "name": "customers.search",
      "arguments": { "q": "Henderson" }
    }
  }'
```

<Callout type="info">
The `Accept: application/json, text/event-stream` header is required. The MCP endpoint uses the streamable HTTP transport and expects this header on every request.
</Callout>

## Error responses

Tool failures return a JSON-RPC error result (not an HTTP error code, except for auth failures). The error `message` is human-readable. Common cases:

| Situation | HTTP status | Error kind |
|---|---|---|
| Missing or invalid token | 401 | — |
| Token expired | 401 | — |
| Insufficient scope | 403 | `insufficient_scope` |
| Input failed validation | 200 (RPC error) | `invalid_input` |
| Record not found | 200 (RPC error) | `not_found` |
| Conflicting state | 200 (RPC error) | `conflict` |
| Rate limited | 429 | — (see `Retry-After` header) |

See [Errors](/developers/errors) and [Rate limits](/developers/rate-limits) for full details.

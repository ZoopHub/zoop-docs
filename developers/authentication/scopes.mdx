Scopes tell Zoop what your integration is allowed to do. When you create an API key or register an OAuth client, you request a set of scopes, and every API call you make is checked against those scopes.

## How scopes work

A scope is a string in the format `verb:resource`. There are two verbs:

- `read:*` ŌĆö fetch or list data
- `write:*` ŌĆö create, update, or delete data

Write access does not imply read access. If your integration needs to update a customer and also fetch that customer back, request both `write:customers` and `read:customers`.

Admin scopes follow a different pattern (`admin:*`) and are covered [below](#admin-scopes).

## Umbrella scopes

Two scopes expand into multiple child scopes automatically. You can request either the umbrella or the specific children ŌĆö the result is the same.

### Catalog umbrella

`read:catalog` expands to `read:catalog_items` + `read:catalog_categories`.  
`write:catalog` expands to `write:catalog_items` + `write:catalog_categories`.

Request `read:catalog` if your integration needs to browse the full service catalog. Request `read:catalog_items` or `read:catalog_categories` alone if you only need one sub-resource.

```text
# Requesting the umbrella ŌĆö gives access to both sub-resources
scope=read:catalog

# Requesting only one sub-resource
scope=read:catalog_items
```

### Estimates ŌĆö legacy alias for quotes

Zoop renamed "estimates" to "quotes" in May 2026. The `read:estimates` and `write:estimates` scopes still work and expand to `read:quotes` and `write:quotes` respectively. They exist so tokens minted before the rename continue to work without reauthorization.

**New integrations should request `read:quotes` / `write:quotes` directly.** The estimates aliases are kept for backward compatibility but are not the preferred form.

| Legacy scope | Expands to |
|---|---|
| `read:estimates` | `read:quotes` |
| `write:estimates` | `write:quotes` |

## Sub-resources under customers

Contacts, locations, and tags are sub-resources of a customer. They do not have their own scopes ŌĆö they are governed by `read:customers` and `write:customers`.

For example, to list or create contacts for a customer, your token must hold `read:customers` or `write:customers`. The same applies to service locations and customer tags.

Notes are different: they have their own scope pair (`read:notes` / `write:notes`) and are independent of `read:customers`.

Both scopes are available to external callers. The notes tools require `read:notes` to list or fetch and `write:notes` to create, update, or archive, and an API key or OAuth token that holds them can use them.

One difference applies to writes: creating, updating, or archiving a note requires a user-actor token, because every note change is attributed to the user who made it. A tenant-actor API key can read notes but cannot author or change them ŌĆö for that you need an OAuth token (or a user-actor API key) carrying `write:notes`.

## Admin scopes

Admin scopes grant elevated access. They can only be requested by integrations authorized by the tenant owner ŌĆö office and technician roles cannot mint tokens that include admin scopes.

<Callout type="warning">
Request admin scopes only when strictly necessary. Most integrations do not need them.
</Callout>

| Scope | What it grants |
|---|---|
| `admin:tenant` | Manage team members and tenant settings |
| `admin:billing` | Access billing and subscription info |
| `admin:credentials` | Manage API keys and OAuth clients |

`admin:credentials` is required to create or revoke API keys via the API. The external access endpoints (`/api/external-access`) enforce this scope.

## User-actor vs tenant-actor

Zoop has two kinds of authenticated actors:

- **User-actor** ŌĆö a human user (owner, office staff, or technician) who authorized your OAuth client. Scopes are further constrained by the user's role. For example, a technician-role user cannot authorize `admin:tenant` even if your client requests it.
- **Tenant-actor** ŌĆö an API key that acts on behalf of the business as a whole, not on behalf of any individual user. API keys are scoped at the time of creation and are not bound to a user's role.

When using OAuth (user-actor), the scopes your token receives are the intersection of what you requested and what the authorizing user's role allows. When using an API key (tenant-actor), the scopes are exactly what was set on the key.

See [OAuth](/developers/authentication/oauth) and [API keys](/developers/authentication/api-keys) for setup details.

## Requesting scopes

Pass scopes as a space-delimited string in the `scope` parameter.

<Tabs>
  <Tab title="OAuth authorization request">
    ```text
    GET /oauth/authorize
      ?client_id=YOUR_CLIENT_ID
      &redirect_uri=https://yourapp.example.com/callback
      &response_type=code
      &scope=read:customers write:customers read:jobs
      &state=RANDOM_STATE_VALUE
    ```
  </Tab>
  <Tab title="API key (scopes set at creation)">
    ```bash
    curl -X POST https://api.zoopapp.com/{tenantId}/api/external-access \
      -H "Authorization: Bearer ADMIN_TOKEN" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "My integration",
        "scopes": ["read:customers", "read:jobs", "write:invoices"]
      }'
    ```
  </Tab>
</Tabs>

## Choosing the right scopes

Request only the scopes your integration actually uses. This limits the blast radius if a token is leaked and makes the OAuth consent screen easier for users to trust.

A typical read-only reporting integration needs:

```text
read:customers read:jobs read:invoices read:quotes
```

A full-service scheduling integration might need:

```text
read:customers write:customers read:jobs write:jobs read:catalog read:quotes write:quotes
```

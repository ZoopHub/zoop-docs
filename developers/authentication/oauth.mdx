OAuth lets your app act on behalf of a Zoop user — reading their customers, creating jobs, or sending invoices — without ever seeing their password. The user approves access through a consent screen; you get a short-lived access token and a refresh token.

Use OAuth when your app serves multiple Zoop accounts or when you need the access to be tied to a specific user identity. If you are building a server-to-server integration that runs unattended, [API keys](/developers/authentication/api-keys) are simpler.

## How it works

Zoop uses **OAuth 2.1** with **PKCE** (Proof Key for Code Exchange). The authorization server is **Supabase** — Zoop owns the consent screen and the grant table that binds scopes to the issued token, but the actual token exchange happens against the Supabase auth endpoint.

The issued access token is an **ES256-signed JWT**. A custom access token hook injects `tenant_id`, `scopes`, and `role` into the claims at issuance. Zoop verifies these claims on every request without a database lookup.

```text
Your app              Supabase auth             Zoop
   |                       |                      |
   |-- GET /authorize -->   |                      |
   |                       |-- 302 → consent -->  |
   |                       |                      |
   |   (user approves)     |                      |
   |                       |                      |
   |-- POST /token ------> |                      |
   |                       |                      |
   |<-- access_token + refresh_token -------------|
   |                                              |
   |-- POST /api/mcp (Bearer <access_token>) ---> |
```

## Prerequisites

Before you can run this flow, you need:

1. **A registered OAuth client.** Contact your Zoop instance admin. They register the client via the Supabase Management API or dashboard. Registration returns a `client_id` — and a `client_secret` for confidential (server-side) clients.

2. **Allowed scopes.** Zoop's scope model is default-deny. A newly registered client cannot request any scopes until a Zoop admin explicitly adds them to the `oauth_client_registrations` table. The consent screen shows exactly those registered scopes, regardless of what you put in the `scope` query parameter.

3. **A pre-registered redirect URI.** The redirect URI your app will receive the authorization code at must be registered with the client. Unregistered URIs are rejected.

<Callout type="info">
If you are building an MCP integration, the OAuth flow is the same. The token you receive can be sent directly to `POST /api/mcp` with an `Authorization: Bearer` header.
</Callout>

## Discovery

Do not hardcode the authorization server URL. Resolve it from the protected-resource metadata document first:

```bash
GET https://app.zoop.pro/.well-known/oauth-protected-resource
```

Response:

```json
{
  "resource": "https://app.zoop.pro",
  "authorization_servers": ["https://<project>.supabase.co/auth/v1"],
  "bearer_methods_supported": ["header"],
  "resource_documentation": "https://docs.zoop.work/api"
}
```

<Callout type="warning">
The `resource` value is environment-specific (production, staging, local). Always read it from the discovery response — do not hardcode `https://app.zoop.pro`. You will need the exact `resource` string when you build the authorization URL and the token request.
</Callout>

Then fetch the Supabase authorization server metadata to get the `authorization_endpoint` and `token_endpoint`:

```bash
GET https://<project>.supabase.co/auth/v1/.well-known/oauth-authorization-server
```

This endpoint is served by Supabase. It returns standard OAuth 2.0 AS metadata including supported grants, PKCE methods, and scopes.

## The authorization flow

<Steps>
  <Step title="Generate PKCE parameters and state">
    PKCE (Proof Key for Code Exchange) prevents authorization code interception. You generate a random secret (`code_verifier`), hash it (`code_challenge`), and send the hash. Later, you prove you hold the original secret by sending the verifier.

    `state` is a random value you bind to the user's session. You verify it in the callback to block cross-site request forgery (CSRF) attacks.

    ```ts
    import { randomBytes, createHash } from 'crypto'

    const codeVerifier = randomBytes(64).toString('base64url')
    const codeChallenge = createHash('sha256').update(codeVerifier).digest('base64url')
    const state = randomBytes(32).toString('base64url')

    // Store both in the user's session — you need them in the callback.
    session.codeVerifier = codeVerifier
    session.oauthState = state
    ```

    Store `codeVerifier` and `state` server-side (signed cookie, Redis session, etc.) — they are needed in steps 3 and 4.
  </Step>

  <Step title="Redirect the user to the authorization endpoint">
    Build the authorization URL and send the user's browser there.

    ```text
    GET https://<project>.supabase.co/auth/v1/oauth/authorize
      ?client_id=<your_client_id>
      &redirect_uri=<your_registered_redirect_uri>
      &response_type=code
      &code_challenge=<codeChallenge>
      &code_challenge_method=S256
      &state=<state>
      &scope=openid
      &resource=https://app.zoop.pro
    ```

    Key parameters:

    <ParamField name="client_id" type="string" required>
      Your registered client ID.
    </ParamField>

    <ParamField name="redirect_uri" type="string" required>
      Must exactly match a URI pre-registered with the client.
    </ParamField>

    <ParamField name="response_type" type="string" required>
      Always `code`.
    </ParamField>

    <ParamField name="code_challenge" type="string" required>
      The base64url-encoded SHA-256 hash of your `code_verifier`.
    </ParamField>

    <ParamField name="code_challenge_method" type="string" required>
      Always `S256`. Plain is not supported.
    </ParamField>

    <ParamField name="state" type="string" required>
      Your CSRF token. Verified in the callback.
    </ParamField>

    <ParamField name="scope" type="string">
      OIDC scopes (`openid`, `email`). Custom Zoop scopes come from the client's registered scope list — what you send here does not change what appears on the consent screen.
    </ParamField>

    <ParamField name="resource" type="string" required>
      The `resource` value from `/.well-known/oauth-protected-resource` (RFC 8707). Required by MCP-spec-compliant clients. Must match exactly.
    </ParamField>

    Supabase validates the parameters and redirects to the Zoop consent landing page:

    ```text
    https://app.zoop.pro/oauth/consent?authorization_id=<opaque_id>
    ```

    If the user has multiple tenants, they pick which one to bind the grant to. If they are not signed in, they are redirected to `/login` first and returned here after.

    <Callout type="warning">
    One grant per (client, user). If a user re-consents for a different tenant, the previous grant is overwritten. The client loses access to the previous tenant and gains access to the new one. There are no parallel per-tenant grants for the same client and user.
    </Callout>
  </Step>

  <Step title="Handle the callback">
    After the user approves, Supabase redirects to your `redirect_uri`:

    ```text
    https://your-app.example.com/callback?code=<authorization_code>&state=<your_state>
    ```

    **Verify `state` before doing anything else.** Compare it to the value you stored in step 1. If they do not match, reject the request — this indicates a CSRF or authorization code injection attempt.

    ```ts
    const { code, state } = new URL(request.url).searchParams

    if (state !== session.oauthState) {
      throw new Error('State mismatch — possible CSRF attack')
    }

    // Safe to proceed.
    ```

    If the user previously granted consent for this client, the consent screen is skipped and the redirect happens immediately.
  </Step>

  <Step title="Exchange the code for tokens">
    Send the authorization code and your `code_verifier` to the token endpoint. The server verifies that the verifier hashes to the `code_challenge` sent in step 2.

    <Tabs>
      <Tab title="Confidential client (server-side)">
        Confidential clients have a `client_secret`. Send it as HTTP Basic auth.

        ```bash
        curl -X POST https://<project>.supabase.co/auth/v1/oauth/token \
          -H "Authorization: Basic $(echo -n '<client_id>:<client_secret>' | base64)" \
          -H "Content-Type: application/x-www-form-urlencoded" \
          --data-urlencode "grant_type=authorization_code" \
          --data-urlencode "code=<authorization_code>" \
          --data-urlencode "redirect_uri=<your_registered_redirect_uri>" \
          --data-urlencode "code_verifier=<codeVerifier>" \
          --data-urlencode "resource=https://app.zoop.pro"
        ```
      </Tab>
      <Tab title="Public client (native app / SPA)">
        Public clients have no `client_secret`. Send `client_id` in the body instead.

        ```bash
        curl -X POST https://<project>.supabase.co/auth/v1/oauth/token \
          -H "Content-Type: application/x-www-form-urlencoded" \
          --data-urlencode "grant_type=authorization_code" \
          --data-urlencode "client_id=<your_client_id>" \
          --data-urlencode "code=<authorization_code>" \
          --data-urlencode "redirect_uri=<your_registered_redirect_uri>" \
          --data-urlencode "code_verifier=<codeVerifier>" \
          --data-urlencode "resource=https://app.zoop.pro"
        ```
      </Tab>
    </Tabs>

    The `resource` parameter (RFC 8707) must appear in both the authorization request (step 2) and this token request.

    Successful response:

    ```json
    {
      "access_token": "<es256-signed-jwt>",
      "token_type": "bearer",
      "expires_in": 3600,
      "refresh_token": "<opaque-refresh-token>"
    }
    ```

    The access token expires in 3600 seconds (1 hour). Store the refresh token securely — you use it to get a new access token without repeating the full flow.
  </Step>

  <Step title="Call the API">
    Send the access token as a `Bearer` token in the `Authorization` header.

    ```bash
    curl -X POST https://app.zoop.pro/api/mcp \
      -H "Authorization: Bearer <access_token>" \
      -H "Content-Type: application/json" \
      -H "Accept: application/json, text/event-stream" \
      -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
    ```

    The `Accept: application/json, text/event-stream` header is required — the MCP endpoint uses the streamable HTTP transport.

    ```ts
    const res = await fetch('https://app.zoop.pro/api/mcp', {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${accessToken}`,
        'Content-Type': 'application/json',
        'Accept': 'application/json, text/event-stream',
      },
      body: JSON.stringify({
        jsonrpc: '2.0',
        id: 1,
        method: 'tools/call',
        params: {
          name: 'customers.search',
          arguments: { q: 'Henderson' },
        },
      }),
    })
    ```
  </Step>
</Steps>

## The access token

The access token is an ES256-signed JWT. You can decode it (e.g. at [jwt.io](https://jwt.io)) to inspect the claims. Zoop's verification logic validates these claims on every request.

```json
{
  "iss": "https://<project>.supabase.co/auth/v1",
  "sub": "<user_uuid>",
  "tenant_id": "<tenant_uuid>",
  "scopes": ["read:customers", "read:jobs"],
  "client_id": "<your_client_id>",
  "role": "owner"
}
```

<ResponseField name="iss" type="string">
  Issuer. Pinned to the Supabase project's auth endpoint. Tokens from any other issuer are rejected.
</ResponseField>

<ResponseField name="sub" type="string">
  The Supabase user UUID of the user who granted consent.
</ResponseField>

<ResponseField name="tenant_id" type="string">
  The UUID of the Zoop tenant the user bound the grant to. All API calls with this token are scoped to this tenant.
</ResponseField>

<ResponseField name="scopes" type="array">
  The scopes granted at consent time. Comes from the `oauth_client_grants` row written when the user approved the consent screen — not from whatever was in the `scope` parameter at authorization.
</ResponseField>

<ResponseField name="client_id" type="string">
  Your OAuth client ID.
</ResponseField>

<ResponseField name="role" type="string">
  The user's role in the tenant: `owner`, `office`, or `tech`. Zoop coerces unrecognized values to `tech` (least-privileged).
</ResponseField>

## Refreshing tokens

When the access token expires, use the refresh token to get a new one without repeating the full browser flow.

<Tabs>
  <Tab title="Confidential client">
    ```bash
    curl -X POST https://<project>.supabase.co/auth/v1/oauth/token \
      -H "Authorization: Basic $(echo -n '<client_id>:<client_secret>' | base64)" \
      -H "Content-Type: application/x-www-form-urlencoded" \
      --data-urlencode "grant_type=refresh_token" \
      --data-urlencode "refresh_token=<refresh_token>" \
      --data-urlencode "resource=https://app.zoop.pro"
    ```
  </Tab>
  <Tab title="Public client">
    ```bash
    curl -X POST https://<project>.supabase.co/auth/v1/oauth/token \
      -H "Content-Type: application/x-www-form-urlencoded" \
      --data-urlencode "grant_type=refresh_token" \
      --data-urlencode "client_id=<your_client_id>" \
      --data-urlencode "refresh_token=<refresh_token>" \
      --data-urlencode "resource=https://app.zoop.pro"
    ```
  </Tab>
</Tabs>

A successful response is the same shape as the original token response — a new `access_token` and `expires_in`, and potentially a new `refresh_token`.

If the refresh request returns `401` with `error: "invalid_token"`, the session has been terminated on the Supabase side (e.g. the user signed out). You must restart the full authorization flow.

## Scopes

Custom Zoop scopes are determined by the client's registration in `oauth_client_registrations`, not by the `scope` parameter in the authorization request. A newly registered client has no custom scopes by default — a Zoop admin must add them.

Common scope sets:

| Use case | Scopes |
|---|---|
| Read-only field service data | `read:customers read:jobs read:invoices read:quotes` |
| Full field service read/write | `read:customers write:customers read:jobs write:jobs read:invoices write:invoices` |
| Catalog management | `read:catalog write:catalog` |

Request only the scopes your integration actually needs. See [scopes](/developers/authentication/scopes) for the full catalog.

## Error reference

| Situation | Status | Body |
|---|---|---|
| Token expired | `401` | `{"error":"expired"}` — use the refresh token |
| OAuth token invalidated (session terminated) | `401` | `{"error":"invalid_token"}` — restart the authorization flow |
| Insufficient scope | `403` | `{"error":"insufficient_scope"}` — the token does not hold the required scope |
| Wrong tenant | `403` | `{"error":"wrong_tenant"}` — the token's `tenant_id` does not match the requested tenant |
| Rate limited | `429` | `Retry-After` header included |
| JWKS endpoint down | `5xx` | Retryable — Zoop rethrows JWKS infrastructure failures so you get a retryable 5xx, not a misleading 401 |

Handle `5xx` responses with exponential backoff. Do not discard tokens on a `5xx` — the token may be valid and the error may be transient.
